# Vision

Este documento registra a visão principal do projeto.

Ele deve responder: o que estamos construindo, para quem, por quê, qual transformação esperamos gerar e qual é o limite da primeira versão.

## Objetivo

Evitar que o projeto comece apenas com telas soltas ou código sem direção.

Antes de implementar, a visão precisa estar clara.

## Regra principal

Todo projeto deve ter uma visão simples o suficiente para qualquer pessoa entender.

Se a visão não estiver clara, não começar pela implementação.

## Frase de visão

Escrever em uma frase:

O projeto existe para ajudar [público] a resolver [problema] através de [solução], gerando [resultado].

Exemplo:

O projeto existe para ajudar restaurantes a venderem direto pelo próprio cardápio digital, organizando pedidos, pagamentos e atendimento em uma central própria.

## Problema

Descrever o problema atual.

Perguntas:

- O que está ruim hoje?
- O que toma tempo?
- O que gera retrabalho?
- O que atrapalha o cliente?
- O que atrapalha a operação?
- O que faz perder dinheiro?
- O que deixa a experiência pouco profissional?

## Público

Definir quem usa.

Possíveis públicos:

- cliente final;
- admin;
- equipe interna;
- gestor;
- vendedor;
- operador;
- financeiro;
- dono do negócio;
- visitante público;
- usuário com link compartilhado.

- Quem é:
- O que precisa fazer:

- Qual problema resolve:
- Qual tela/fluxo usa:

## Solução

Explicar a solução sem tecnicismo excessivo.

Perguntas:

- O que o sistema faz?
- Qual é o fluxo principal?
- Como melhora a experiência?
- Como melhora a operação?
- Como reduz retrabalho?
- Como gera mais controle?
- Como pode evoluir no futuro?

## Resultado esperado

Definir o que precisa acontecer para o projeto ser considerado bem-sucedido.

Exemplos:

- cliente entende melhor o serviço;
- equipe acompanha status;
- pedidos ficam organizados;
- pagamentos ficam registrados;
- cliente acessa atualizações sozinho;
- visual fica mais profissional;
- operação ganha controle;
- sistema reduz dependência de planilha/WhatsApp.

## Fora do escopo inicial

Registrar o que não será feito agora.

## Futuro

Registrar ideias futuras sem misturar com o MVP.

Exemplos:

- pagamentos;
- WhatsApp automático;
- relatórios;
- app mobile;
- multiusuário;
- multiunidade;
- automações;
- IA;
- analytics.

## Riscos

Registrar riscos iniciais:

- prazo curto;
- dependência de API;
- falta de identidade visual;
- dados incompletos;
- integração complexa;
- segurança de links;
- login;
- escopo grande demais;
- dependência de decisão do cliente.
